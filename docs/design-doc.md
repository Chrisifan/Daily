# Daily 设计文档

## 1. 文档目的

本文档描述 Daily 当前版本的产品定位、信息架构、核心界面设计方向，以及与当前代码仓库结构对应的实现边界。文档基于 2026-04-15 的实际项目结构维护，不再沿用早期 `smart-workbench / ai-service` 的单体目录假设。

## 2. 当前项目结构

当前仓库采用 pnpm monorepo 结构：

```text
Daily/
├── packages/
│   ├── web/                  # React + TypeScript + Vite 前端
│   │   └── src/
│   │       ├── app/          # 路由与应用布局
│   │       ├── application/  # 预留的业务编排层目录
│   │       ├── domain/       # 核心实体类型
│   │       ├── features/     # 页面级功能模块
│   │       ├── i18n/         # 国际化初始化
│   │       ├── shared/       # 通用 hooks / ui / utils / services
│   │       ├── storage/      # schema / seeds / repositories
│   │       └── locales/      # i18n 文案
│   ├── desktop/              # Tauri 2 桌面容器（Rust）
│   │   └── src/
│   └── service/              # Express + LangChain 服务
│       └── src/
├── docs/
└── showcase/
```

这意味着产品文档中的“桌面应用”在实现上被拆成三个独立部分：

- `packages/web`：负责 UI、交互、页面状态与本地视图模型
- `packages/desktop`：负责窗口、原生权限、Tauri 桥接
- `packages/service`：负责邮件、位置、分类与建议等服务端能力

## 3. 产品定位

Daily 是一个面向知识工作者的桌面工作台，围绕三个核心问题构建：

- 今天有哪些关键事项需要被看见
- 现在最值得进入哪个任务上下文
- 不同工作区的信息如何被统一组织

产品不是一个泛后台，也不是聊天式 AI 外壳，而是一个“当天工作视图 + 工作区切换器 + 智能建议层”。

## 4. 目标用户

- 同时推进多个项目、多个身份或多个交付线的个人工作者
- 需要在日程、邮件、任务和上下文之间频繁切换的团队成员
- 对效率工具、沉浸式工作台和可视化工作流有持续需求的用户

## 5. 当前一级信息架构

当前代码仓库中已存在或已预留的一级页面结构如下：

1. 首页 Home
2. 日程 Schedule
3. 工作区 Workspaces
4. 集成 Integrations
5. 设置 Settings

补充说明：

- `Inbox` 相关实体与 mock 数据已存在，但独立页面仍处于预留/演进中
- `Onboarding` 已有启动引导弹层
- “智能助手”当前更接近 service 层能力，而不是独立聊天页

## 6. 当前页面职责

### 6.1 首页

首页是“当天工作视图”，而不是功能导航页。当前结构由四块组成：

- 主卡：天气、时间、问候、今日摘要
- 右上摘要卡：日程、项目、邮件、工作区等小型统计入口
- 今日待办卡：最多展示少量关键日程，突出接下来要处理的事项
- 今日总览与工作区卡：展示系统建议和当前推荐工作区

首页当前强调：

- 当日优先
- 低操作成本
- 从“看见信息”快速进入“进入上下文”

### 6.2 日程页

日程页是当前最核心、也最成熟的功能页之一。当前重点为单日时间轴体验：

- 24 小时纵向时间线
- 半小时刻度
- 快速创建、编辑、选择优先级、关联工作区
- 空白时间点击创建
- 冲突与间隔关系的可视化表达

### 6.3 工作区页

工作区页当前已经收敛为极简卡片列表，而不是厚重的项目仪表盘：

- 每张卡片只保留图标、标题、描述、类型 badge、状态 badge
- 视觉语言与首页保持一致的浅色实体卡片
- 页面主操作为右上角的“新建工作区”按钮

### 6.4 集成页

集成页承载未来外部系统接入入口，包括：

- 邮件
- 日历
- 位置 / 天气
- 其他第三方能力

### 6.5 设置页

设置页当前用于管理：

- 主题与显示偏好
- 天气 / 定位配置
- 后续权限与同步偏好

## 7. 当前设计原则

### 7.1 当日优先

默认先回答“今天”和“现在”，而不是罗列系统全部信息。

### 7.2 智能但克制

智能能力用于排序、归类、补全和建议，不替代主操作流。

### 7.3 工作区清晰

任务、邮件、日程尽量能够落到具体工作区，降低上下文混乱。

### 7.4 桌面沉浸感

界面更像工作台而不是网页后台，强调卡片节奏、状态层次和环境感。

### 7.5 轻量交互优先

信息提示、计数、状态摘要要和真正的操作按钮视觉上清晰区分，避免界面把“信息”伪装成“动作”。

## 8. 当前视觉与交互约束

这些约束已经在实现中落地，并应继续作为设计基线：

- 主题以浅色实体卡片为主，不再依赖大面积玻璃态白字
- 全局颜色、圆角、间距尽量复用 `packages/web/src/App.css` 中的共享 token
- 按钮、badge、tips、状态标签必须区分语义
- 表单控件和单行触发器优先统一为 32px 视觉高度
- 所有创建、编辑、连接、配置类表单统一使用弹窗/模态层承载，不以内联页面区块直接展开
- 所有删除类危险操作必须先经过显式确认弹窗，不能在第一次点击时直接执行
- 所有异步操作按钮都必须提供明确反馈：执行中显示 loading，成功后显示 success toast，失败后显示 error toast，且执行期间禁止重复点击
- 优先级语义颜色必须复用共享 token，不在页面局部重新定义
- 首页“今日待办”中的关联工作区信息以轻量标签展示，不占据主内容流
- 集成账号的元数据必须进入应用数据库；密码、授权码、token 等敏感凭据不得写入 JSON、localStorage 或普通 SQLite 字段，必须进入系统安全存储

## 9. 核心数据实体

当前前端 domain 层的核心实体包括：

- `Workspace`
- `ScheduleItem`
- `InboxItem`
- `TaskItem`
- `WeatherSnapshot`

其中：

- 所有时间统一以 UTC 存储
- UI 展示在本地时区转换
- `ScheduleItem` 可关联 `workspaceId`
- `InboxItem` 可携带 `workspaceTypeHint` 与建议动作

## 10. 当前已落地能力

截至当前代码结构，已落地或已具备可运行基础的能力包括：

- React + Vite + Tauri 2 的桌面工作台壳
- 首页概览与天气感知主卡
- 日程页时间轴与弹窗编辑
- 工作区列表与详情页基础展示
- 设置弹层与主题设置基础能力
- i18n 双语文案结构
- mock 数据驱动的本地原型
- AI service 的邮件、分类、建议、位置接口骨架
- IMAP 邮件解析与同步服务基础能力，位于 `packages/service/src/services/email/`

## 11. 当前技术边界

### 11.1 Web

`packages/web` 负责：

- 所有页面 UI
- 本地视图模型与交互状态
- mock 数据与本地原型流程
- 调用 desktop / service 的桥接入口

### 11.2 Desktop

`packages/desktop` 负责：

- Tauri 窗口能力
- 原生权限与能力桥接
- 桌面运行时打包

### 11.3 Service

`packages/service` 负责：

- `/api/classify`
- `/api/suggestions`
- `/api/email`
- `/api/location`

它承载对 LangChain、OpenAI、IMAP 邮件同步等服务端能力的统一出口。

## 12. 当前版本重点

当前产品最重要的设计重点仍然是：

1. 首页可读性和节奏
2. 日程视图与日程卡片的易读性
3. 工作区上下文的轻量承载
4. 外部信息接入后的信息归位

## 13. 下一阶段设计方向

下一阶段建议继续推进：

- 首页信息层次和工作流衔接优化
- 日程与工作区的双向关联体验
- 收件/邮件摘要的独立页面化
- 集成页中的账号连接、状态反馈与断连处理
- service 返回结果与前端 domain 模型的进一步对齐

## 14. 文档维护原则

后续设计文档更新应遵循以下规则：

- 目录结构必须以 `packages/*` 现状为准
- 已撤销的设计方案不应继续作为“当前方案”描述
- 设计文档中的路径必须引用真实存在的目录或文件
- 产品设计、技术设计与局部 spec/plan 之间的结构表述必须保持一致
