# 智能工作台桌面应用设计文档

## 1. 产品定位

智能工作台是一款面向知识工作者的桌面应用，围绕"当天该做什么、现在最该关注什么、不同工作区如何切换"三个核心问题，提供统一入口。

产品目标：

- 把天气、日程、工作区、邮件、日历、待办等分散信息收敛到一个可操作的桌面工作台。
- 降低用户在多个应用之间切换的成本。
- 通过智能聚合与提醒能力，帮助用户更快进入工作状态。

目标用户：

- 需要同时处理多个项目或身份的个人工作者
- 需要在邮件、会议、任务和文档之间频繁切换的团队成员
- 对效率工具、工作台、自定义工作流有较强需求的用户

## 2. 核心设计原则

- 当日优先：默认先展示"今天"和"现在"，而不是把所有信息平铺出来。
- 智能但克制：智能能力为决策和组织服务，不打断主流程。
- 工作区清晰：所有信息都可以落到具体工作区，避免上下文混乱。
- 系统融合：逐步接入系统日历、邮件和本地能力，而不是把应用做成信息孤岛。
- 桌面沉浸感：界面更像一个"工作场"，而不是网页式后台。

## 3. 信息架构

### 3.1 一级结构

1. 首页概览
2. 日程总览
3. 工作区
4. 收件与通知
5. 智能助手
6. 设置与集成

### 3.2 首页概览包含内容

- 与当前位置天气相关的动态背景
- 今日日期、当前时段、天气摘要
- 当日日程时间轴
- 高优先级事项提醒
- 工作区快捷入口
- 今日收件摘要
- 日历冲突或空档提示

### 3.3 工作区包含内容

- 已有工作区列表
- 新建工作区
- 工作区状态概览
- 与工作区绑定的项目、文件、日程、邮件与快捷动作

## 4. 关键功能拓展

### 4.1 首页概览

在你提出的基础上，首页建议不仅展示"今天的日程卡片"，还应提供以下增强能力：

- 今日时间轴：按小时展示会议、专注时段、截止时间与提醒
- 重要性分层：使用优先级和影响范围区分普通事项、关键事项、风险事项
- 当前焦点模块：只突出"接下来 90 分钟最重要的事"
- 日程总览入口：支持切换为日、周、月视图
- 空档推荐：当检测到时间空档时，推荐处理待办、回信或进入深度工作

### 4.2 天气感知界面

首页背景不只是装饰，建议让天气与工作节奏联动：

- 晴天：光感强、对比清晰、节奏轻快
- 阴天：柔和雾面材质、降低视觉刺激
- 雨天：动态雨幕、玻璃质感、强调安静专注
- 夜晚：低照度背景、增强当前任务聚焦
- 极端天气：加入提示，例如出行会议提醒、通勤建议

天气信息可驱动：

- 背景动效
- 色板切换
- 出行类日程提示
- 智能建议文案

### 4.3 日程系统

建议建立统一的"事件模型"，让不同来源的内容都能转成标准日程：

- 基础字段：标题、开始时间、结束时间、重要性、来源、所属工作区、地点、备注
- 高级字段：是否可移动、是否需要准备材料、是否与他人协作、是否与邮件关联

日程能力应包括：

- 手动创建、编辑、拖拽调整
- 按来源过滤：系统日历、邮件提取、手动创建、工作区计划
- 冲突检测：会议时间重叠、准备时间不足、通勤风险
- 智能补全：从会议标题、邮件正文中自动识别地点、参与者、材料
- 一键转任务：把日程中的行动项转为待办

### 4.4 邮件与系统日历集成

邮件和日历是首页信息的核心输入源，建议以"聚合、理解、建议"为主线接入，而不是第一版就直接控制外部系统。

邮件接入分阶段方案：

第一阶段：

- 接入 Gmail、Outlook Mail 或 IMAP 邮箱
- 拉取未读邮件、今日邮件、带附件邮件
- 生成发件人、主题、摘要、时间等基础卡片
- 在首页展示与今日任务相关的邮件摘要

第二阶段：

- 从邮件中提取会议时间、地点、截止时间、行动项
- 自动识别邮件更适合归属哪个工作区类型
- 对可执行邮件生成建议动作，例如"加入日程""加入待办""归档到工作区"

第三阶段：**邮件智能分析与日程生成**

这是邮件接入的核心能力阶段，在完成邮件读取和工作区归属后，系统进一步理解邮件内容，自动生成结构化的日程建议。

##### 邮件内容分析引擎

分析引擎从邮件正文中提取以下结构化实体：

- **时间信息**：会议时间、截止时间、预约时间、时间范围（含时区）
- **地点信息**：会议室、线上会议链接（Zoom/Teams/腾讯会议等）、物理地址
- **参与者**：邮件中明确提到的姓名或邮箱
- **行动项**：以 "请回复""请确认""请在 XX 前完成""待办""action item" 等触发词引导的任务
- **关联上下文**：项目名称、任务编号、相关文档链接

技术实现建议：
- 使用 LLM 对邮件正文进行结构化抽取，输出标准化 JSON
- 维护一份时间表达式正则库（如 "下周二下午 3 点""June 15th"）做快速预解析，减少 LLM 调用成本
- 对常见会议邀请格式（如 .ics 附件、Outlook HTML 格式）做专项解析，直接提取标准字段

##### 日程价值判断

并非所有包含时间信息的邮件都值得生成日程。系统需判断邮件内容的"日程价值"：

**高价值（建议生成日程）**：
- 明确的时间 + 地点 + 参与者，如会议邀请、面试邀约
- 明确的截止时间 + 任务描述，如 "请在 XX 前完成 XX"
- 预约类邮件，如医生预约、餐厅预约、服务预约
- 事件通知类，如航班确认、火车票、酒店预订

**中价值（可能生成日程）**：
- 有时间但缺少地点或参与者的模糊邀约
- 有截止时间但任务不明确
- 讨论类邮件中某人提出的时间建议

**低价值（不生成日程）**：
- 自动通知类（系统更新、广告、订阅）
- 已过期的邀请或截止时间
- 仅提及"尽快""最近"等模糊时间

判断依据可包括：邮件来源（是否为日历邀请）、触发词匹配、实体完整度（时间+地点+参与者是否同时存在）、用户历史行为（是否曾将同类邮件转为日程）。

##### 日程建议生成

通过分析引擎输出，系统生成结构化的日程建议 `ScheduleSuggestion`：

```
ScheduleSuggestion:
  - suggestedTitle: string          // 建议日程标题（可编辑）
  - startTime: DateTime             // 开始时间
  - endTime: DateTime               // 结束时间（默认 1 小时，可根据上下文估算）
  - location: string | null         // 地点或会议链接
  - attendees: string[]              // 提取的参与者邮箱/姓名
  - importance: high | medium | low  // 重要性等级
  - sourceEmailId: string            // 来源邮件 ID
  - sourceEmailSubject: string      // 来源邮件标题（方便回溯）
  - triggerType: meeting | deadline | appointment | reminder | discussion
  - actionItems: string[]            // 邮件中提取的行动项列表
  - confidence: number (0-1)        // 分析置信度
  - reasoning: string               // 生成理由，如"检测到明确会议时间2026年4月15日14:00及腾讯会议链接"
```

建议标题生成示例：
- 原邮件："关于 Q2 产品规划的讨论，明天下午 3 点腾讯会议"
- 建议标题："产品规划讨论"（可用户编辑）

##### 用户审批流程

日程生成默认不直接写入日历，而是以"建议"形式呈现，由用户确认或忽略：

- **建议模式（默认）**：新邮件分析后，在首页或工作区以悬浮卡片形式展示日程建议，用户点击"加入日历"确认或"忽略"拒绝
- **自动添加模式（可选）**：用户可在设置中开启高频场景（如下午日程邀请）的自动添加，经 n 次成功学习后可启用
- **一键审批**：同一人发送的同类邀约（如固定站会），可标记为"以后自动添加"，减少重复确认

审批 UI 要求：
- 建议卡片需清晰展示：日程标题、时间、地点、来源邮件摘要
- 操作成本低：单次点击完成确认或忽略
- 可快捷编辑后添加（如修改标题或时间）
- 支持"查看原邮件"跳转，防止误判

##### 邮件与日程关联

生成的日程与原始邮件之间建立双向关联：

- **邮件 → 日程**：在邮件详情页显示"已生成日程"标签及日程卡片入口
- **日程 → 邮件**：在日程详情页显示"来源邮件"入口，点击跳转至原始邮件
- **同步更新**：若原始邮件被撤回或日程被修改，相关联的一方可发出提醒或自动更新
- **历史追溯**：用户可通过邮件搜索找到由其生成的所有日程，也可通过日历找到所有来自邮件的日程

### 4.10 邮件驱动的日程生成工作流

本节描述邮件从接收 → 分析 → 建议 → 审批 → 入日历的完整自动化工作流。

#### 完整工作流步骤

```
邮件接收 → 邮件解析 → 内容分析 → 日程价值判断 → 日程建议生成 → 用户审批 → 入日历 → 邮件关联
```

**Step 1：邮件接收与解析**
- 通过 Gmail / Graph / IMAP 连接器拉取新邮件
- 提取标准字段：发件人、收件人、主题、正文（HTML / 纯文本）、时间戳、附件、Message-ID、Thread-ID
- 若邮件包含 .ics 格式邀请附件，优先解析标准 iCal 字段

**Step 2：内容分析与结构化抽取**
- 将邮件正文输入 EmailAnalyzer，提取时间、地点、参与者、行动项等实体
- 对标准日历邀请格式直接走快速通道，跳过 LLM 解析
- 输出 EmailAnalysisResult（含提取实体 + 置信度）

**Step 3：日程价值判断**
- 根据触发类型、实体完整度、来源可信度计算日程价值分数
- 高于阈值 → 生成 ScheduleSuggestion
- 低于阈值 → 仅归档至收件箱，不生成建议

**Step 4：日程建议生成与展示**
- 生成结构化建议，存入本地建议表
- 在首页/工作区/收件摘要卡中展示待确认建议（带醒目角标）
- 支持气泡推送或系统通知（取决于用户设置）

**Step 5：用户审批**
- 用户确认 → 创建正式 ScheduleItem，关联来源邮件
- 用户忽略 → 建议标记为已处理，不再次提示（除非同主题新邮件触发）
- 用户编辑 → 弹出简化编辑框，保存后以编辑结果创建日程

**Step 6：入日历与关联**
- 将 ScheduleItem 通过 CalendarConnector 写入目标日历
- 记录 `relatedEmailIds` 字段，建立双向关联
- 若写入失败（如权限不足），保留建议记录并提示用户手动处理

#### 关键设计原则

- **用户主导**：日程从不由系统静默自动生成，必须经用户确认（除非用户主动开启特定白名单场景）
- **可追溯**：用户任何时候都能找到某日程对应的原始邮件
- **低打扰**：建议以非侵入方式展示，不打断当前工作
- **可学习**：用户对建议的接受/忽略行为逐步影响后续判断准确率
- **容错**：分析失败时不展示错误，只静默记录日志，保证主流程不受影响

#### 与工作区的联动

- 系统自动判断邮件所属工作区，日程建议同步带上工作区标签
- 用户确认日程时，可一键切换工作区归属
- 工作区详情页显示"来自邮件的待处理日程建议"列表



工作区不是普通文件夹，而是信息上下文容器。首页上的日程、邮件、文件、待办都应该尽可能能归属于某个工作区。

工作区通用能力：

- 有名称、类型、颜色、状态和当前目标
- 能绑定邮件、日历、文件夹、任务和联系人
- 能展示最近活动、当前重点和下一步建议
- 能通过模板快速创建

工作区视图建议具备：

- 列表视图：快速浏览所有工作区
- 看板视图：按状态区分活跃、待跟进、归档
- 详情页：显示该工作区的时间线、相关邮件、待办、会议和资料
- 智能摘要：自动概括本周进展、风险和下一步动作

### 4.6 工作区类型模板

根据当前需求，建议把工作区预设为四类：

#### 4.6.1 代码工作区

适合内容：

- 开发任务
- bug 修复
- code review
- API 联调
- 技术会议

默认模块：

- 今日编码任务
- Git 分支 / PR / 提交摘要
- 相关 issue
- 技术邮件
- 会议安排
- 本地项目入口

智能建议：

- 从邮件识别 review 请求、bug 反馈、上线通知
- 从日历识别 standup、评审、发版会议
- 汇总今天改动的模块和阻塞项

#### 4.6.2 图片处理工作区

适合内容：

- 设计稿处理
- 图片修图
- 素材整理
- 批量导出
- 视觉评审

默认模块：

- 当前素材集
- 导出任务
- 图片规格
- 附件邮件
- 设计评审日程
- 最近文件

智能建议：

- 从邮件提取附件和修改要求
- 根据日历安排生成会前导出清单
- 跟踪版本变化和交付状态

#### 4.6.3 写作工作区

适合内容：

- 文章
- 文案
- 提纲
- 周报
- 调研整理

默认模块：

- 当前稿件
- 提纲结构
- 截止时间
- 参考资料
- 修改意见邮件
- 采访或讨论安排

智能建议：

- 从邮件提取修改意见
- 自动生成今日写作计划
- 自动总结本周写作进度

#### 4.6.4 通用工作区

适合内容：

- 临时项目
- 行政事务
- 个人安排
- 跨类型任务

默认模块：

- 今日事项
- 日历安排
- 收件摘要
- 快捷入口

### 4.7 工作区自动归类规则

为了减少手动整理成本，建议加入轻量级工作区分类策略：

- 邮件标题或正文包含 PR、commit、bug、接口、发版
  默认归类为代码工作区
- 邮件包含 PSD、海报、导出、尺寸、图片、素材
  默认归类为图片处理工作区
- 邮件包含提纲、文章、润色、文案、周报、稿件
  默认归类为写作工作区
- 无法准确判断时
  进入通用工作区或等待用户确认

### 4.8 收件与通知中心

为了让首页不被打扰，建议把通知集中管理：

- 邮件更新
- 日历变更
- 工作区提醒
- 智能建议
- 系统级事件，例如网络异常或同步失败

通知支持：

- 仅提醒高优先级
- 批量稍后处理
- 直接转成任务或加入某个工作区

### 4.9 智能助手

智能工作台的"智能"部分建议围绕具体工作流，而不是做一个泛聊天入口。

可落地能力：

- 今天帮我排一下优先级
- 根据邮件内容生成日程建议
- 帮我把本周会议整理成工作区摘要
- 检测今天是否存在冲突和空档
- 根据当前天气和会议地点给出出发建议
- 为某个工作区生成周报摘要

## 5. 首页 UI 方案

### 5.1 布局建议

- 顶部：轻量品牌区与全局操作
- 中央：一张主卡展示当前时间、天气和核心概览入口
- 右侧：四张小卡展示日程、工作区、邮件、日历摘要
- 底部：一张今日待办卡和一张今日总览 / 工作区卡

### 5.2 首页首屏模块

1. 天气驱动背景
2. 当前时间与问候主卡
3. 小型摘要卡：日程 / 工作区 / 邮件 / 日历
4. 今日待办卡
5. 总览与工作区卡

### 5.3 视觉风格建议

- 整体风格：偏桌面原生、冷静、沉浸、轻玻璃材质
- 天气背景：大面积氛围层配合少量粒子或流动效果
- 颜色策略：主色不固定，由天气与时段驱动
- 字体策略：标题有识别度，正文偏工具感和可读性

## 6. 关键页面建议

### 6.1 日程总览

## 8. 集成架构与同步策略

### 8.1 集成层结构

建议采用统一接入层，把不同服务的差异屏蔽掉：

- Connector 层
  负责调用 Gmail、Graph、Google Calendar、系统日历等外部接口
- Normalizer 层
  把外部数据转成统一的 InboxItem 和 ScheduleItem
- Router 层
  根据内容把数据归属到工作区类型或具体工作区
- Suggestion 层
  生成"加入待办""加入日程""绑定工作区"等建议动作

**新增 EmailAnalyzer 层**

在 Connector 与 Normalizer 之间新增专用分析层，负责邮件内容的深度理解：

```
邮件流：Connector → EmailAnalyzer → Normalizer → Router → Suggestion
```

EmailAnalyzer 职责：
- 接收原始邮件（发件人、主题、正文、附件列表）
- 解析标准 .ics 日历邀请格式，提取标准事件字段
- 对普通邮件正文调用 LLM 结构化抽取，输出 EmailAnalysisResult
- 做日程价值判断，决定是否生成 ScheduleSuggestion
- 输出结构化分析结果，传递给下游 Normalizer 和 Suggestion 层

EmailAnalyzer 内部模块：
- **ICSParser**：专门解析 .ics 格式日历邀请，快速路径，零 LLM 调用
- **EntityExtractor**：基于 LLM 的通用邮件内容抽取（时间/地点/参与者/行动项）
- **ValueScorer**：根据实体完整度、触发词匹配、历史行为计算日程价值分数
- **SuggestionGenerator**：将分析结果组装为标准 ScheduleSuggestion 结构

设计原则：
- EmailAnalyzer 为独立可插拔模块，不绑定特定 LLM 提供商
- 分析结果带置信度，低置信度时下游建议层降级处理（如仅提示"有邮件提到时间"而不生成具体日程）
- 分析异步执行，新邮件进入队列后后台处理，不阻塞邮件列表展示

### 8.2 同步策略

建议采用三种同步模式：

- 启动同步
  应用打开时拉取今日关键数据
- 前台增量同步
  每隔一段时间同步邮件与日历摘要
- 手动强制同步
  用户点击刷新时立即同步

**邮件监听与增量分析策略**

邮件同步在常规同步模式基础上，增加以下策略：

- **新邮件实时检测**：通过 Gmail/Graph push notification（webhook）或定期轮询（间隔 60s）检测新邮件
- **邮件分级同步**：
  - 第一层（立即）：同步邮件头部（发件人、主题、时间戳），用于列表展示
  - 第二层（按需）：按需拉取邮件正文，用于分析引擎输入
  - 第三层（异步）：后台触发 EmailAnalyzer 对新邮件进行日程价值分析
- **增量分析策略**：
  - 新邮件进入分析队列，FIFO 顺序处理
  - 每封邮件的分析结果独立存储（EmailAnalysisResult），支持重复分析
  - 已分析过的邮件若收到新回复，自动追加到分析队列
  - 分析超时或失败时，记录错误并标记，不阻塞后续邮件
- **分析频率控制**：
  - 避免对短时间内大量新邮件同时触发 LLM 分析
  - 可配置分析速率限制（如每分钟最多分析 5 封邮件）
  - 低优先级时段（如深夜）降低分析频率
- **已读/未读状态同步**：邮件的已读状态变化时，不重新触发分析（除非是线程追加）

推荐策略：

- 首页只读取今日和本周的数据
- 邮件先同步头部信息和摘要，再按需展开正文
- 日历先同步最近 7 天和未来 14 天

### 8.3 错误与状态设计

系统应能显示：

- 已连接
- 同步中
- 授权失效
- 服务暂不可用
- 最近同步时间

### 8.4 隐私与权限

- 邮件正文和日历内容默认只用于本地工作台理解
- 所有接入都应有单独授权说明
- 用户可关闭自动分类和智能建议
- 重要数据应支持本地清除和断开连接

## 9. 桌面端能力建议

桌面应用相比 Web 版更适合做这些能力：

- 系统通知

- 本地日历读取
- 开机启动
- 全局快捷键唤起
- 离线缓存与最近数据可用
- 本地文件快速打开

- 桌面容器：Electron 或 Tauri
- UI：React 或原型阶段先用静态页面
- 本地数据：SQLite 或 IndexedDB
- 同步层：统一同步服务，屏蔽不同来源差异

当前最适合的落地策略是：

- 首页做天气驱动的工作现场
- 日历做第一优先级接入源
- 邮件做第二优先级接入源
- 工作区以代码、图片处理、写作为核心模板
- 智能层负责归类、建议和冲突识别

## 13. 总结

基于你现在的两个起点，这个智能工作台最适合发展的方向是：

- 首页做"今天的工作现场"
- 工作区做"上下文容器"
- 邮件和日历做"信息输入源"
- 智能能力做"理解、整理、提醒和推荐"

这样产品会有明确主线，也能从原型逐步演进到真正可用的桌面工作平台。
